Help Me Choose
Apple Ecosystem Advisory Recommendation Tool
Product Requirements Document
Portfolio Edition
01 Executive Summary

Help Me Choose 是一款面向 Apple 线上选购场景的顾问式决策工具。它通过少量高价值问题，帮助用户从“我已经知道要买哪条产品线”过渡到“我可以更有把握地锁定具体 SKU”。

项目并不是对传统参数对比页的平移，而是尝试以更接近顾问的方式，输出包含主推款、备选方案、推荐原因 (Why) 与配件联动的完整决策结果。核心目标是降低认知负荷与选择焦虑，并缩短探索到购买的路径。

The Problem
  • Apple 官网的问题不是信息不够，而是决策路径不清晰。
  • 传统参数对比页很难回答用户最在意的问题：“为什么是这款，而不是另一款？”
  • 预算敏感型用户容易产生“系统默认在把我往更贵方案推”的防御心理。
Target Audience
  • 核心画像：18–35 岁，具备 Apple 购买意图，但在型号决策上犹豫的用户。
  • 重点覆盖：首次生态入门用户、跨代升级用户、预算高度敏感用户。
Product Design Principles

少题高价值 (High ROI Questions)

避免冗长问卷，每一道问题都必须对最终 SKU 的收敛产生真实影响。

心态优先于价格 (Mindset > Price)

先识别购买心态与硬需求，再引入预算维度，降低“被推销”的主观感受。

品类专属链路 (Category Customization)

iPhone / MacBook / AirPods 采用不同决策路径，而不是套用同一套模板。

决策可解释性 (Explainable Output)

结果页不仅展示商品，还必须明确说明推荐动因，让结果更可信。

02 Key Interface Nodes
Mockup 01

欢迎页

以一句明确的价值主张和一个强 CTA 降低进入门槛，让用户快速理解这是一个帮助选对款的工具，而不是参数列表。

Mockup 02

品类选择

不替用户做宽泛决定，而是确认其已知起点。这一设计减少了“我明明已经想买 iPhone，你却还在问我要不要买手机”的无效摩擦。

Mockup 03

升级顾问问题

通过“你现在在用什么手机”切入 iPhone 路径，使系统从预算筛选器转向更符合真实心智的升级顾问逻辑。

Mockup 04

结果页闭环

结果页同时输出主推、备选、搭配建议与下一步动作，用于完成最后的决策确认，而不是把问题重新交回商品详情页。

03 Core Logic & Architecture

A. 推荐逻辑框架 (Recommendation Logic)

Key Takeaway：系统的推荐逻辑并不是“预算越高越往上推”，而是先排除明显不适合的选项，再按品类进入更符合真实决策心智的路径，最后通过 trade-off 逻辑生成主推与备选。

Example 01｜首次入门 + 控制预算

输入：Android / 第一次用 iPhone + 想先进入苹果生态 + 控制预算

输出：iPhone 16 128GB 作为主推；原因不在于“最低配”，而在于“进入生态的最稳妥入口”。

Example 02｜升级用户 + 想换更明显体验

输入：iPhone 14–15 Pro 用户 + 希望获得更明显升级感 + 愿意一步到位

输出：更高阶路径优先，重点考察续航、大屏与完整体验，而不是机械地回到标准版。

Example 03｜AirPods 的硬需求优先

输入：用户连续表达“降噪”诉求

输出：系统必须让降噪能力拥有更高优先级，避免普通款出现在主推位。

Example 04｜中间档不是妥协，而是自然答案

输入：想要更大屏、更长续航，但不想直接上 Pro

输出：Plus 路径存在的意义，是提供一个真实、合理的中间档选择。

B. 结果页信息层级 (Result Page Structure)

Key Takeaway：结果页承担的是决策收拢，而不是商品展示。Why 文案负责建立信任，备选款提供理性退路，配件区承接完整方案与商业价值，CTA 则支持进一步比较、保存与确认。
04 User Feedback & Key Insights

“怎么总是推同一台？”

首轮测试中，最直接的反馈是系统过于容易给出“Apple 体系内的默认答案”，而没有先理解用户这次究竟想怎么花这笔钱。

设计动作：加入预算状态 / 购买心态

V3 引入“控制预算 / 中间档 / 愿意多花 / 一步到位”等选项，使预算敏感型用户能够显式表达自己的购买方式。

“iPhone 的关键不是预算，而是值不值得换。”

后续反馈进一步指出，用户在 iPhone 上更关心“我现在用什么”“为什么要换”“这次升级值不值”，而不是单纯价格区间。

设计动作：从预算筛选器转向升级顾问

iPhone 路径逐步演进为“当前机型 → 换机原因 → 购买方式”，这也是整个项目差异化最明确的一次升级。

“连续选降噪，却给我推普通款。”

AirPods 测试暴露出一个典型问题：预算权重不应该压过强需求，尤其是降噪这种明确功能诉求。

设计动作：强需求提权

后续逻辑明确将降噪、尺寸、续航、影像等强需求前置，从根源上减少“系统并不理解我”的感受。

05 Version Evolution
V1

最小可运行原型

完成欢迎页、问卷与结果页闭环，证明这一方向可以被演示、被讨论、被继续迭代。

V3

预算状态成为显式变量

加入预算状态后，系统不再只输出体系内的“稳妥标准答案”，而开始识别不同用户的购买心态。

V5–V6

iPhone 升级顾问成型

iPhone 从统一预算筛选升级为更接近真实换机决策的顾问路径，并单独处理首次入门用户。

V7–V8.2

SKU 语义与中间档修正

项目开始统一在售 SKU 与历史参考方案的表达，并引入更自然的中间档路径，使结果更贴近真实购买逻辑。

Evolution Summary：这个项目的价值不只在于最终页面完成度，而在于每一轮迭代都更接近真实用户的选择逻辑：先修问题定义，再修规则结构，最后修商品语义与 trade-off 表达。
06 Metrics & Capability Signal
Success Metrics
Guided Flow Completion Rate
从进入问卷到到达结果页的完成率，用于判断流程是否顺畅。
Recommendation CTR
点击查看主推商品详情的比例，用于衡量推荐结果是否足够有吸引力。
Accessory Attach Rate
联动配件区的点击率，用于衡量完整方案对商业价值的贡献。
Compare Click-through Rate
点击“对比主推与备选”的比例，用于观察用户仍有多大不确定性。
Restart Rate
点击“重新来过”的比例，用于判断推荐是否常常偏离用户预期。
Product Capabilities Reflected In This PRD
  • 问题定义能力：抓住“已知品类、未知型号”的真实线上决策痛点。
  • 产品抽象能力：把复杂商品矩阵收敛成少量高价值问题与规则结构。
  • 反馈迭代能力：根据真实试用反馈持续修改规则，而不是停留在页面优化层面。
  • Trade-off 能力：平衡预算、硬需求、升级价值与商业目标，而非默认推荐最贵方案。
  • 表达能力：把推荐结果写成可解释、可信的 Why 文案，而不是纯参数列表。
07 Next Steps
Roadmap
  • 继续细化 iPhone 升级树：把“当前机型 → 换机动因”的颗粒度做得更真实。
  • 优化中间档推荐：为 Plus 与同系列低存储版本建立更自然的进入路径。
  • 统一 SKU 语义：在当前在售与历史参考方案之间保持一致表达。
  • 继续扩大外部测试：重点观察首次入门用户与预算敏感用户的理解路径。
Closing View

Help Me Choose 的核心价值不在于覆盖所有商品，而在于系统是否真正理解了用户的犹豫与取舍。

这个项目最重要的，并不是给用户一个答案，而是让用户相信：为什么这个答案适合自己。
